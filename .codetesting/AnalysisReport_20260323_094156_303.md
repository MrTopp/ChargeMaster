## ChargeMaster.UnitTests.csproj - ChargeMaster.Services.Shelly.UnitTests.ShellyMqttServiceTests.SetupAsync_WhenConnectAsyncFails_PropagatesException
- **Confidence**: High
- **Test File**: ChargeMaster.UnitTests\Services\Shelly\ShellyMqttServiceTests.cs
- **Bug Location**: ChargeMaster\Services\Shelly\ShellyMqttService.cs@196-197

### Analysis
The test cannot be properly fixed without production code changes. The ShellyMqttService creates IMqttClient directly in ConnectAsync (line 196-197) using 'new MqttClientFactory().CreateMqttClient()', making it impossible to mock the MQTT client for testing. Additionally, MQTTnet library doesn't reliably throw exceptions on connection failures to unreachable brokers. The test attempts to verify exception propagation from ConnectAsync, but this behavior is not testable without dependency injection support. While InitiateTemperatures does catch exceptions (lines 149-183), the core issue is that ConnectAsync cannot be mocked to reliably produce testable failure scenarios.

### Suggested Fix
To make this code testable, refactor ShellyMqttService to support dependency injection of IMqttClient. Add a constructor parameter or property for IMqttClient:

```csharp
private readonly IMqttClient? _mqttClient;

public ShellyMqttService(
    IServiceScopeFactory serviceScopeFactory,
    ILogger<ShellyMqttService> logger,
    IMqttClient? mqttClient = null)
{
    // ... existing initialization ...
    _mqttClient = mqttClient;
}

public async Task ConnectAsync(string brokerAddress, int brokerPort = 1883, string? clientId = null)
{
    clientId ??= $"chargemaster-{Guid.NewGuid().ToString("N")[..8]}";
    
    // Only create new client if not injected
    if (_mqttClient == null)
    {
        var factory = new MqttClientFactory();
        _mqttClient = factory.CreateMqttClient();
    }
    
    // ... rest of the method ...
}
```

This would allow tests to inject a mocked IMqttClient that can reliably simulate connection failures, making the test scenario properly testable.

## ChargeMaster.UnitTests.csproj - ChargeMaster.Services.Shelly.UnitTests.ShellyMqttServiceTests.TemperatureChanged_MultipleSubscribers_AllReceiveIdenticalInitialValues
- **Confidence**: High
- **Test File**: ChargeMaster.UnitTests\Services\Shelly\ShellyMqttServiceTests.cs
- **Bug Location**: ChargeMaster\Services\Shelly\ShellyMqttService.cs@68

### Analysis
The production code has a bug in the TemperatureChanged event's add accessor. When a new subscriber is added, it invokes `_temperatureChangedHandlers` (the entire multicast delegate containing all subscribers) three times, causing all existing subscribers to receive duplicate events. This is why subscriber1 receives 6 events instead of 3 - it gets 3 events when it first subscribes, then 3 more when subscriber2 subscribes. The fix is to invoke only the newly added handler (`value` parameter) with the initial values, not all handlers.

### Suggested Fix
In the `TemperatureChanged` event's `add` accessor (lines 68-70 in ShellyMqttService.cs), change from invoking `_temperatureChangedHandlers` to invoking only the newly added `value` handler:

```csharp
value?.Invoke(this, new ShellyTemperatureChangedEventArgs("arbetsrum", Temperatures["arbetsrum"]));
value?.Invoke(this, new ShellyTemperatureChangedEventArgs("hall", Temperatures["hall"]));
value?.Invoke(this, new ShellyTemperatureChangedEventArgs("sovrum", Temperatures["sovrum"]));
```

This ensures that only the newly subscribed handler receives the initial temperature values, not all existing subscribers.
